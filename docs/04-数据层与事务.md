# 数据层与事务

## MyBatis-Plus + MySQL

### 依赖

```xml
<dependency>
    <groupId>com.baomidou</groupId>
    <artifactId>mybatis-plus-spring-boot3-starter</artifactId>
    <version>3.5.15</version>
</dependency>
<dependency>
    <groupId>com.mysql</groupId>
    <artifactId>mysql-connector-j</artifactId>
</dependency>
```

> Spring Boot 3.x 必须用 `mybatis-plus-spring-boot3-starter`，不能用 `mybatis-plus-boot-starter`（那是给 2.x 的）。

### 配置文件

```yaml
spring:
  datasource:
    url: jdbc:mysql://IP:3306/demo?useUnicode=true&characterEncoding=utf-8&serverTimezone=Asia/Shanghai
    username: root
    password: '密码'
```

### 实体类

```java
@TableName("user")                          // 映射到哪张表
public class User {

    @TableId(type = IdType.AUTO)            // 主键自增
    private Long id;
    private String name;                    // 驼峰命名自动映射到下划线字段
    private String email;                   // email → email 列
    private Integer age;
    private LocalDateTime createTime;       // createTime → create_time 列
}
```

### Mapper 接口

```java
@Mapper
public interface UserMapper extends BaseMapper<User> {
    // 3 行代码，零 SQL，CRUD 全有
}
```

### BaseMapper 自带方法

| 方法 | 作用 |
|------|------|
| `selectList(null)` | 查全部，`null` = 无条件 |
| `selectById(id)` | 按主键查 |
| `insert(entity)` | 新增 |
| `updateById(entity)` | 按主键更新 |
| `deleteById(id)` | 按主键删除 |

### 驼峰自动映射

数据库字段 `create_time` → Java 字段 `createTime`，MyBatis-Plus 会自动转换，不需要手动配置。

## Service 层 — 业务逻辑与事务

### 三层架构

```
Controller → 接收请求，调 Service，返回响应（不包含业务逻辑）
    ↓
Service    → 处理业务逻辑，调用 Mapper（可调多个 Mapper）
    ↓
Mapper     → 只跟数据库交互，一行 SQL 一个方法
```

### Controller vs Service vs Mapper

| | Controller | Service | Mapper |
|---|---|---|---|
| 职责 | 接收请求、参数校验 | 业务规则、流程编排 | 只跟数据库打交道 |
| 注解 | `@RestController` | `@Service` | `@Mapper` |
| 调用 | 调 Service | 调 Mapper（或其他 Service） | 只写 SQL |
| 事务 | 不管 | `@Transactional` | 不管 |

### @Transactional — 事务

**只有抛异常时才回滚。你不抛，它就认为成功。**

```java
@Transactional
public void transfer(Long fromId, Long toId, BigDecimal amount) {
    int rows = accountMapper.deduct(fromId, amount);
    if (rows == 0) {
        throw new IllegalArgumentException("余额不足或账户不存在");  // 抛异常 → 回滚
    }
    int rows2 = accountMapper.addBalance(toId, amount);
    if (rows2 == 0) {
        throw new IllegalArgumentException("目标账户不存在");       // 抛异常 → 回滚
    }
}
```

流程：

```
1. deduct 扣钱 → SQL 执行成功
2. addBalance 失败 → 返回 0 行
3. throw 异常
4. @Transactional 拦截到异常 → 把第 1 步的扣钱撤销
5. 数据回到调用前的状态
```

### 必须自己写 SQL 的场景

`BaseMapper.updateById` 是"你给什么值就更新成什么"，但转账需要在**当前值基础上加减**，所以必须自己写：

```java
// Mapper 里自己写 SQL
@Update("UPDATE account SET balance = balance - #{amount} WHERE id = #{id} AND balance >= #{amount}")
int deduct(@Param("id") Long id, @Param("amount") BigDecimal amount);
```

### 常见坑

- **未检查返回值**：`UPDATE` 执行成功但可能更新 0 行（条件不满足），必须检查返回值并抛异常，否则 `@Transactional` 不会回滚
- **checked exception 不回滚**：`@Transactional` 默认只回滚 `RuntimeException`，`IOException` 等 checked exception 需要额外指定 `@Transactional(rollbackFor = Exception.class)`

## 事务回滚的底层原理

### 一句话概括

Spring 在你方法执行前关闭了数据库的**自动提交**，方法正常结束就提交，抛异常就回滚。

### 第 0 层：数据库默认是"自动提交"

默认情况下，每执行一条 SQL，MySQL 立刻提交。没有反悔机会：

```sql
UPDATE account SET balance = balance - 100 WHERE id = 1;  -- 立刻生效，改不回去了
UPDATE account SET balance = balance + 100 WHERE id = 2;  -- 立刻生效
```

### 第 1 层：JDBC 提供了手动控制事务的能力

```java
Connection conn = dataSource.getConnection();
conn.setAutoCommit(false);  // 关闭自动提交 ← 关键

try {
    // 执行两条 SQL...
    conn.commit();           // 成功 → 提交
} catch (Exception e) {
    conn.rollback();         // 异常 → 回滚
}
```

Spring 的 `@Transactional` 本质就是帮你做了这件事。

### 第 2 层：Spring AOP 代理拦截方法

你写的类被 Spring 包了一层**代理对象**，代理在调用你的方法前后织入了事务逻辑：

```
你调用 transfer()
    ↓
代理对象拦截：
  1. 获取数据库连接
  2. conn.setAutoCommit(false)     ← 关闭自动提交
  3. 调用你的 transfer() 方法       ← 你的业务逻辑
  4. 没抛异常 → conn.commit()      ← 提交
     抛异常了 → conn.rollback()     ← 回滚
  5. 归还连接
```

所以你的 `transfer()` 里没有任何事务代码——代理帮你包好了。

### 第 3 层：回滚到底回滚了什么

**SQL 已经执行了，数据库里的数据已经被改了，但还没"定稿"。**

时间线：

```
T1: 事务开始，autoCommit=false
T2: UPDATE account SET balance = balance - 100 WHERE id = 1
    → MySQL 确实执行了这条 SQL
    → Buffer Pool（内存）里 id=1 的 balance：500 → 400
    → 同时写下 undo log，记住："id=1 原值是 500"
T3: 抛异常
T4: rollback → MySQL 用 undo log 把 Buffer Pool 里的 balance 改回 500
```

**不是"什么都没发生"，而是"发生了但撤销了"。** 证明：同一事务内，第一条 SQL 的结果对第二条 SQL 是可见的——如果第一条真的"没执行"，第二条的计算基础就错了。

### 其他事务看到的还是旧数据（MVCC）

```
              事务内操作                  回滚
你的事务:       500 → 400      undo log →  400 → 500
其他连接看到的:  500              500          500（始终看不到中间态）
```

MySQL 的 MVCC 保证：你未 commit 的修改，别人永远看不到。

## 什么时候不该加 @Transactional

事务不是免费的，有代价：**持有一个数据库连接 + 可能持有行锁，直到方法结束才释放。**

### 1. 只有一次写操作

```java
// ❌ 不需要
@Transactional
public void createUser(User user) {
    userMapper.insert(user);  // 一条 SQL，MySQL 本身就能保证原子性
}
```

### 2. 纯读操作

```java
// ❌ 不需要
@Transactional
public User getUser(Long id) {
    return userMapper.selectById(id);  // 不修改数据，回滚无意义
}
```

### 3. 调了外部服务

```java
// ❌ 灾难
@Transactional
public void register(User user) {
    userMapper.insert(user);          // 写数据库
    smsService.send(user.getPhone()); // 发短信（耗时 2 秒）
}
// 发短信的 2 秒里，数据库连接和行锁一直不释放，别人全排队
```

### 决策表

| 场景 | 加事务？ |
|------|---------|
| 多张表写入，需要同时成功 | ✅ 加 |
| 同一张表多个 UPDATE/DELETE，有逻辑关联 | ✅ 加 |
| 单条 INSERT/UPDATE/DELETE | ❌ 不加 |
| 纯 SELECT | ❌ 不加 |
| 方法里调了外部服务（短信、API） | ❌ 不加 |
