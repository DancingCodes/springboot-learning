## Service 层 — 业务逻辑与事务

### 三层架构

```
Controller → 接收请求，调 Service，返回响应（不包含业务逻辑）
    ↓
Service    → 处理业务逻辑，调用 Mapper（可调多个 Mapper）
    ↓
Mapper     → 只跟数据库交互，一行 SQL 一个方法
```

### Controller vs Service vs Mapper

| | Controller | Service | Mapper |
|---|---|---|---|
| 职责 | 接收请求、参数校验 | 业务规则、流程编排 | 只跟数据库打交道 |
| 注解 | `@RestController` | `@Service` | `@Mapper` |
| 调用 | 调 Service | 调 Mapper（或其他 Service） | 只写 SQL |
| 事务 | 不管 | `@Transactional` | 不管 |

### @Transactional — 事务

**只有抛异常时才回滚。你不抛，它就认为成功。**

```java
@Transactional
public void transfer(Long fromId, Long toId, BigDecimal amount) {
    int rows = accountMapper.deduct(fromId, amount);
    if (rows == 0) {
        throw new IllegalArgumentException("余额不足或账户不存在");  // 抛异常 → 回滚
    }
    int rows2 = accountMapper.addBalance(toId, amount);
    if (rows2 == 0) {
        throw new IllegalArgumentException("目标账户不存在");       // 抛异常 → 回滚
    }
}
```

流程：

```
1. deduct 扣钱 → SQL 执行成功
2. addBalance 失败 → 返回 0 行
3. throw 异常
4. @Transactional 拦截到异常 → 把第 1 步的扣钱撤销
5. 数据回到调用前的状态
```

### 必须自己写 SQL 的场景

`BaseMapper.updateById` 是"你给什么值就更新成什么"，但转账需要在**当前值基础上加减**，调整余额需要**精确覆盖**，所以必须自己写：

```java
// Mapper 里自己写 SQL
@Update("UPDATE account SET balance = balance - #{amount} WHERE id = #{id} AND balance >= #{amount}")
int deduct(@Param("id") Long id, @Param("amount") BigDecimal amount);

@Update("UPDATE account SET balance = balance + #{amount} WHERE id = #{id}")
int addBalance(@Param("id") Long id, @Param("amount") BigDecimal amount);

@Update("UPDATE account SET balance = #{amount} WHERE id = #{id}")
int setBalance(@Param("id") Long id, @Param("amount") BigDecimal amount);
```

| 方法 | SQL | 用途 |
|------|-----|------|
| `deduct` | `balance = balance - #{amount}` | 扣款，带 `WHERE balance >= #{amount}` 防超扣 |
| `addBalance` | `balance = balance + #{amount}` | 加钱 |
| `setBalance` | `balance = #{amount}` | 直接设置余额（调整功能） |

对应的 Service 方法：

```java
// adjustBalance：直接设置余额，不需要 @Transactional（单条 UPDATE）
public void adjustBalance(Long id, BigDecimal amount) {
    if (amount.compareTo(BigDecimal.ZERO) < 0) {
        throw new IllegalArgumentException("余额不能为负数");
    }
    int rows = accountMapper.setBalance(id, amount);
    if (rows == 0) {
        throw new IllegalArgumentException("账户不存在");
    }
}

// transfer：两条 UPDATE，需要 @Transactional 保证原子性
@Transactional
public void transfer(Long fromId, Long toId, BigDecimal amount) {
    int rows = accountMapper.deduct(fromId, amount);
    if (rows == 0) throw new IllegalArgumentException("余额不足或账户不存在");
    int rows2 = accountMapper.addBalance(toId, amount);
    if (rows2 == 0) throw new IllegalArgumentException("目标账户不存在");
}
```

`adjustBalance` 不需要事务（单条 SQL），`transfer` 需要（两条 SQL 必须同时成功）。

### 常见坑

- **未检查返回值**：`UPDATE` 执行成功但可能更新 0 行（条件不满足），必须检查返回值并抛异常，否则 `@Transactional` 不会回滚
- **checked exception 不回滚**：`@Transactional` 默认只回滚 `RuntimeException`，`IOException` 等 checked exception 需要额外指定 `@Transactional(rollbackFor = Exception.class)`

## 事务回滚的底层原理

### 一句话概括

Spring 在你方法执行前关闭了数据库的**自动提交**，方法正常结束就提交，抛异常就回滚。

### 第 0 层：数据库默认是"自动提交"

默认情况下，每执行一条 SQL，MySQL 立刻提交。没有反悔机会：

```sql
UPDATE account SET balance = balance - 100 WHERE id = 1;  -- 立刻生效，改不回去了
UPDATE account SET balance = balance + 100 WHERE id = 2;  -- 立刻生效
```

### 第 1 层：JDBC 提供了手动控制事务的能力

```java
Connection conn = dataSource.getConnection();
conn.setAutoCommit(false);  // 关闭自动提交 ← 关键

try {
    // 执行两条 SQL...
    conn.commit();           // 成功 → 提交
} catch (Exception e) {
    conn.rollback();         // 异常 → 回滚
}
```

Spring 的 `@Transactional` 本质就是帮你做了这件事。

### 第 2 层：Spring AOP 代理拦截方法

你写的类被 Spring 包了一层**代理对象**，代理在调用你的方法前后织入了事务逻辑：

```
你调用 transfer()
    ↓
代理对象拦截：
  1. 获取数据库连接
  2. conn.setAutoCommit(false)     ← 关闭自动提交
  3. 调用你的 transfer() 方法       ← 你的业务逻辑
  4. 没抛异常 → conn.commit()      ← 提交
     抛异常了 → conn.rollback()     ← 回滚
  5. 归还连接
```

所以你的 `transfer()` 里没有任何事务代码——代理帮你包好了。

### 第 3 层：回滚到底回滚了什么

**SQL 已经执行了，数据库里的数据已经被改了，但还没"定稿"。**

时间线：

```
T1: 事务开始，autoCommit=false
T2: UPDATE account SET balance = balance - 100 WHERE id = 1
    → MySQL 确实执行了这条 SQL
    → Buffer Pool（内存）里 id=1 的 balance：500 → 400
    → 同时写下 undo log，记住："id=1 原值是 500"
T3: 抛异常
T4: rollback → MySQL 用 undo log 把 Buffer Pool 里的 balance 改回 500
```

**不是"什么都没发生"，而是"发生了但撤销了"。** 证明：同一事务内，第一条 SQL 的结果对第二条 SQL 是可见的——如果第一条真的"没执行"，第二条的计算基础就错了。

### 其他事务看到的还是旧数据（MVCC）

```
              事务内操作                  回滚
你的事务:       500 → 400      undo log →  400 → 500
其他连接看到的:  500              500          500（始终看不到中间态）
```

MySQL 的 MVCC 保证：你未 commit 的修改，别人永远看不到。

## 什么时候不该加 @Transactional

事务不是免费的，有代价：**持有一个数据库连接 + 可能持有行锁，直到方法结束才释放。**

### 1. 只有一次写操作

```java
// ❌ 不需要
@Transactional
public void updateUser(User user) {
    userMapper.updateById(user);  // 一条 SQL，MySQL 本身就能保证原子性
}
```

再看一个需要事务的例子——新增用户的同时创建账户：

```java
// ✅ 需要 @Transactional — 两张表写入，必须同时成功
@Transactional
public void add(User user) {
    user.setCreateTime(LocalDateTime.now());
    userMapper.insert(user);                    // user 表
    Account account = new Account();
    account.setUserId(user.getId());
    account.setName(user.getName() + "的账户");
    account.setBalance(BigDecimal.ZERO);
    accountMapper.insert(account);              // account 表
}
```

如果 user 插入成功但 account 插入失败，没有 `@Transactional` 就会产生孤儿用户（没有账户）。有了事务，account 失败时 user 也自动回滚。

### 2. 纯读操作

```java
// ❌ 不需要
@Transactional
public User getUser(Long id) {
    return userMapper.selectById(id);  // 不修改数据，回滚无意义
}
```

### 3. 调了外部服务

```java
// ❌ 灾难
@Transactional
public void register(User user) {
    userMapper.insert(user);          // 写数据库
    smsService.send(user.getPhone()); // 发短信（耗时 2 秒）
}
// 发短信的 2 秒里，数据库连接和行锁一直不释放，别人全排队
```

### 决策表

| 场景 | 加事务？ |
|------|---------|
| 多张表写入，需要同时成功 | ✅ 加 |
| 同一张表多个 UPDATE/DELETE，有逻辑关联 | ✅ 加 |
| 单条 INSERT/UPDATE/DELETE | ❌ 不加 |
| 纯 SELECT | ❌ 不加 |
| 方法里调了外部服务（短信、API） | ❌ 不加 |

## MyBatis-Plus 分页

### 配置分页插件

MyBatis-Plus 分页是插件机制，需要先注册拦截器：

```java
@Configuration
public class MybatisPlusConfig {
    @Bean
    public MybatisPlusInterceptor mybatisPlusInterceptor() {
        return new MybatisPlusInterceptor();
    }
}
```

> MyBatis-Plus 3.5.15 中分页能力已内置到 `MybatisPlusInterceptor` 自身，无需（也无法）再注册 `PaginationInnerInterceptor`。
```

### 使用分页

Service 层创建 `Page` 对象，传给 Mapper 的 `selectPage`：

```java
public Page<User> page(int pageNum, int pageSize) {
    return userMapper.selectPage(new Page<>(pageNum, pageSize), null);
    //                                                          ↑ Wrapper，null = 无条件
}
```

`Page<T>` 对象包含：

| 字段 | 类型 | 说明 |
|------|------|------|
| records | List\<T\> | 当前页数据 |
| total | long | 总记录数 |
| size | long | 每页条数 |
| current | long | 当前页码 |
| pages | long | 总页数 |

### 底层原理

MyBatis-Plus 分页插件拦截 `selectPage` 调用，自动做两件事：

1. 在你写的 SQL 外面包一层 `SELECT count(*) FROM (...)` 拿到总条数
2. 在 SQL 末尾拼接 `LIMIT offset, size` 拿到当前页数据

你不需要手写 COUNT 查询和 LIMIT。
