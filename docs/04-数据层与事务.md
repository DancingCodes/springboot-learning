# 数据层与事务

## MyBatis-Plus + MySQL

### 依赖

```xml
<dependency>
    <groupId>com.baomidou</groupId>
    <artifactId>mybatis-plus-spring-boot3-starter</artifactId>
    <version>3.5.15</version>
</dependency>
<dependency>
    <groupId>com.mysql</groupId>
    <artifactId>mysql-connector-j</artifactId>
</dependency>
```

> Spring Boot 3.x 必须用 `mybatis-plus-spring-boot3-starter`，不能用 `mybatis-plus-boot-starter`（那是给 2.x 的）。

### 配置文件

```yaml
spring:
  datasource:
    url: jdbc:mysql://IP:3306/springbootlearning?useUnicode=true&characterEncoding=utf-8&serverTimezone=Asia/Shanghai
    username: root
    password: '密码'
```

### 连接池（HikariCP）— 自动配置

Spring Boot 引入 `mysql-connector-j` 后，**HikariCP 连接池自动生效**，无需额外依赖和配置。

```
启动时：
  1. Spring Boot 发现 classpath 上有 mysql-connector-j
  2. 读取 spring.datasource.url / username / password
  3. 自动创建 HikariCP 连接池，预建 10 个数据库连接
  4. 请求来了从池里借连接，用完还回去，不销毁
```

没有连接池的话，每次 SQL 请求都要重新 TCP 三次握手 → 执行 → 四次挥手，几十毫秒就浪费在建立连接上。池化后连接复用，只有第一次取连接有开销。

**验证**：启动日志搜 `HikariPool-1 - Starting...`，始终可见。

**默认配置**（不写也能跑，生产按需调）：

| 参数 | 默认值 | 说明 |
|------|------|------|
| `maximumPoolSize` | 10 | 最大连接数 |
| `minimumIdle` | 10 | 最少空闲连接数 |
| `connectionTimeout` | 30000ms | 等待连接超时 |
| `idleTimeout` | 600000ms | 空闲连接过期 |
| `maxLifetime` | 1800000ms | 连接最大存活时间 |

**为什么 MySQL 不用像 Redis 那样加 `commons-pool2`**：
- HikariCP 是 Spring Boot 内置的，`spring-boot-starter-data-jdbc` 已包含
- Lettuce（Redis 客户端）默认单连接多路复用，不池化。`commons-pool2` 给它补齐连接池能力

## 三层架构

```
Controller → 接收请求，调 Service，返回响应（不含业务逻辑）
    ↓
Service    → 处理业务逻辑，调用 Mapper
    ↓
Mapper     → 只跟数据库交互
```

| | Controller | Service | Mapper |
|---|---|---|---|
| 职责 | 接收请求 | 业务逻辑 | 数据库操作 |
| 注解 | `@RestController` | `@Service` | `@Mapper` |
| 事务 | 不管 | `@Transactional` | 不管 |

## @Transactional — 事务

**抛异常才回滚，不抛就提交。**

```java
import org.springframework.stereotype.Service;
import org.springframework.transaction.annotation.Transactional;
import java.math.BigDecimal;

@Service
public class AccountService {

    private final AccountMapper accountMapper;

    public AccountService(AccountMapper accountMapper) {
        this.accountMapper = accountMapper;
    }

    @Transactional
    public void transfer(Long fromId, Long toId, BigDecimal amount) {
        int rows = accountMapper.deduct(fromId, amount);
        if (rows == 0) {
            throw new IllegalArgumentException("余额不足或账户不存在");  // 抛异常 → 回滚
        }
        int rows2 = accountMapper.addBalance(toId, amount);
        if (rows2 == 0) {
            throw new IllegalArgumentException("目标账户不存在");       // 抛异常 → 回滚
        }
    }
}
```

流程：

```
1. deduct 扣钱 → SQL 执行成功
2. addBalance 失败 → 返回 0 行
3. throw 异常 → @Transactional 拦截到 → 回滚第 1 步
4. 数据回到调用前状态
```

### 底层原理

MySQL 默认每条 SQL 自动提交。Spring `@Transactional` 帮你关了自动提交，方法正常结束才提交，抛异常就回滚：

```java
import java.sql.Connection;
import javax.sql.DataSource;

// @Transactional 底层相当于：
Connection conn = dataSource.getConnection();
conn.setAutoCommit(false);  // 关闭自动提交
try {
    // 你的业务代码
    conn.commit();           // 成功 → 提交
} catch (Exception e) {
    conn.rollback();         // 异常 → 回滚
}
```

回滚靠的是 MySQL 的 undo log：SQL 确实执行了，数据在内存里也改了，但 undo log 记下了旧值。rollback 时用 undo log 把数据改回去。未提交的修改，其他事务看不到（MVCC）。

### 必须自己写 SQL 的场景

`BaseMapper.updateById` 是全覆盖更新，转账需要在当前值上加减，得自己写：

```java
import org.apache.ibatis.annotations.Mapper;
import org.apache.ibatis.annotations.Param;
import org.apache.ibatis.annotations.Update;
import java.math.BigDecimal;

@Mapper
public interface AccountMapper extends BaseMapper<Account> {

    @Update("UPDATE account SET balance = balance - #{amount} WHERE id = #{id} AND balance >= #{amount}")
    int deduct(@Param("id") Long id, @Param("amount") BigDecimal amount);

    @Update("UPDATE account SET balance = balance + #{amount} WHERE id = #{id}")
    int addBalance(@Param("id") Long id, @Param("amount") BigDecimal amount);

    @Update("UPDATE account SET balance = #{amount} WHERE id = #{id}")
    int setBalance(@Param("id") Long id, @Param("amount") BigDecimal amount);
}
```

| 方法 | SQL | 用途 |
|------|-----|------|
| `deduct` | `balance = balance - #{amount}` | 扣款，`WHERE balance >= #{amount}` 防超扣 |
| `addBalance` | `balance = balance + #{amount}` | 加钱 |
| `setBalance` | `balance = #{amount}` | 直接设置余额 |

### 什么时候不该加 @Transactional

单条 SQL 不需要（MySQL 本身保证原子性），纯读不需要，方法里调了外部服务（短信、API）也不加（会长时间占着数据库连接）。

| 场景 | 加事务？ |
|------|---------|
| 多张表写入，需要同时成功 | ✅ 加 |
| 同一张表多个 UPDATE/DELETE | ✅ 加 |
| 单条 INSERT/UPDATE/DELETE | ❌ 不加 |
| 纯 SELECT | ❌ 不加 |
| 方法里调了外部服务 | ❌ 不加 |

### 常见坑

- **未检查返回值**：`UPDATE` 可能更新 0 行（条件不满足），必须检查并抛异常，否则不会回滚
- **checked exception 不回滚**：默认只回滚 `RuntimeException`，`IOException` 等需要 `@Transactional(rollbackFor = Exception.class)`

## MyBatis-Plus 分页

### 配置分页插件

```java
import com.baomidou.mybatisplus.extension.plugins.MybatisPlusInterceptor;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;

@Configuration
public class MybatisPlusConfig {

    @Bean
    public MybatisPlusInterceptor mybatisPlusInterceptor() {
        return new MybatisPlusInterceptor();
    }
}
```

### 使用分页

```java
import com.baomidou.mybatisplus.extension.plugins.pagination.Page;

public Page<User> page(int pageNum, int pageSize) {
    return userMapper.selectPage(new Page<>(pageNum, pageSize), null);
}
```

`Page<T>` 包含：

| 字段 | 说明 |
|------|------|
| records | 当前页数据 |
| total | 总记录数 |
| size | 每页条数 |
| current | 当前页码 |
| pages | 总页数 |

分页插件会自动做：包一层 `COUNT(*)` 拿总数 + 拼 `LIMIT` 拿当前页。你不需要手写。

## VO 与 DTO — 实体之外的数据对象

### 为什么需要

Entity 和数据库表一一对应，但真实场景中：

- **出参**往往要组合多张表的数据（比如用户列表同时显示账户余额）
- **入参**只应暴露前端该填的字段（不暴露 id、createTime 等后端控制的字段）

把这些需求都塞进 Entity 会让实体臃肿、职责不清。解决方案是用 VO 和 DTO 各司其职：

```
        入参 DTO               出参 VO
  前端 → UserSaveDTO → Controller → UserVO → 前端
              │                   ↑
              ↓                   │
          User 实体 ──── Service ──┘
              │
              ↓
           Mapper → DB
```

### VO（View Object，视图对象）— 出参用

组合多个表的数据，只包含展示需要的字段：

```java
@Data
public class UserVO {
    private Long id;
    private String name;
    private String email;
    private Integer age;
    private String avatar;
    private LocalDateTime createTime;
    private Long accountId;    // 来自 account 表
    private BigDecimal balance; // 来自 account 表
}
```

Service 中 Mapper 一条 LEFT JOIN SQL 直接查出 VO，无需多次查询和手动组装：

```java
// UserMapper.java
@Select("SELECT u.id, u.name, u.email, u.age, u.avatar, u.create_time, " +
        "a.id AS account_id, a.balance " +
        "FROM user u LEFT JOIN account a ON u.id = a.user_id " +
        "ORDER BY u.id DESC")
Page<UserVO> selectUserPage(Page<UserVO> page);

// UserService.java —— 一行搞定
public Page<UserVO> page(int current, int size) {
    return userMapper.selectUserPage(new Page<>(current, size));
}
```

LEFT JOIN 在数据库层一次查出两张表的数据，MyBatis-Plus 自动映射到 VO 字段，比"先查 user 再批量查 account 再 Java 组装"简洁得多。

### DTO（Data Transfer Object，数据传输对象）— 入参用

只暴露前端该填的字段，加校验注解：

```java
@Data
public class UserSaveDTO {
    private Long id;           // 修改时传，新增时不传

    @NotBlank(message = "姓名不能为空")
    private String name;

    @Email(message = "邮箱格式不正确")
    private String email;

    @Min(0) @Max(150)
    private Integer age;

    private String avatar;
}
```

Controller 中手动 set 转换 DTO → Entity：

```java
@PostMapping
public Result<Void> add(@Valid @RequestBody UserSaveDTO dto) {
    User user = new User();
    user.setName(dto.getName());
    user.setEmail(dto.getEmail());
    user.setAge(dto.getAge());
    user.setAvatar(dto.getAvatar());
    userService.add(user);
    return Result.success(null);
}
```

### 总结

| 对象 | 全称 | 方向 | 职责 |
|------|------|------|------|
| Entity | — | ↔ 持久层 | 和数据库表一一映射，字段 = 列 |
| VO | View Object | ← 出参 | 组合多个表的数据，给前端展示 |
| DTO | Data Transfer Object | → 入参 | 限定前端能传的字段，加校验 |

**核心原则：Entity 只映射自己的表，VO 管展示，DTO 管输入，各司其职。**
