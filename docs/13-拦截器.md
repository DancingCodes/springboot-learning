# 拦截器

## Filter vs Interceptor

| | Filter | Interceptor |
|------|--------|-------------|
| 层级 | Servlet 容器 | Spring 容器 |
| 能注入 Spring Bean | 需要包装（构造注入可行但不直接） | 直接 `@Autowired` |
| 知道执行的 Controller | 不知道 | `handler` 就是目标方法 |
| 能拿到响应内容 | 只能 wrap | `postHandle` 拿到状态码和 ModelAndView |
| 时机 | 请求进入 Servlet 之前 | 请求到达 Controller 前 → Controller 后 → 视图渲染后 |

**简单记**：Filter 管"能不能进"（认证），Interceptor 管"进去之后干什么"（日志、权限、审计）。

## 三个时机

```java
public class LogInterceptor implements HandlerInterceptor {

    // 1. Controller 执行前
    public boolean preHandle(request, response, handler) {
        request.setAttribute("startTime", System.currentTimeMillis());
        log.info("{} {} — handler: {}", method, uri, handler);
        return true; // true=放行，false=拦截
    }

    // 2. Controller 执行后，视图渲染前
    public void postHandle(request, response, handler, modelAndView) {
        log.info("状态码: {}", response.getStatus());
    }

    // 3. 视图渲染后（无论如何都会执行，类似 finally）
    public void afterCompletion(request, response, handler, ex) {
        long elapsed = now - startTime;
        log.info("耗时: {}ms", elapsed);
        if (ex != null) log.error("异常: {}", ex.getMessage());
    }
}
```

**三个阶段的区别**：
- `preHandle` 返回 `false` 会终止请求，后面的所有拦截器和 Controller 都不执行
- `postHandle` 能拿到 ModelAndView（但 REST 接口通常为 null）
- `afterCompletion` **无论如何都会执行**，即使抛异常也有 `ex` 参数

## 注册

```java
@Configuration
public class WebMvcConfig implements WebMvcConfigurer {
    @Override
    public void addInterceptors(InterceptorRegistry registry) {
        registry.addInterceptor(new LogInterceptor())
                .addPathPatterns("/**")           // 拦截所有
                .excludePathPatterns("/doc.html", "/v3/api-docs/**", "/webjars/**"); // 排除
    }
}
```

**注意**：Interceptor 不是 `@Component`，直接 `new` 就行。因为注册时手动 `new` 了，Spring 不会托管它的生命周期。如果需要注入 Bean，可以用 `@Autowired` 注入到 `WebMvcConfig` 里再传进去，或者给 Interceptor 加 `@Component` 注解。

## 和 @Aspect 对比

你现在项目里两种都有：
- **AOP**（`ControllerLogAspect`）— 切的是方法，能拿到参数值、返回值、异常
- **Interceptor**（`LogInterceptor`）— 切的是 HTTP 请求，能拿到 request/response

AOP 比拦截器更细粒度（可以切 Service 任意方法），拦截器只作用于 Controller。但拦截器的优势是能在 `afterCompletion` 做**保证执行的清理工作**，AOP 的 `@After` 不一定能拿到 response 状态码。

实际项目中通常：Filter（认证）→ Interceptor（日志/审计）→ AOP（业务日志/事务）。
