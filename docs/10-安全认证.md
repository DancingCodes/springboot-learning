# Spring Security 安全认证

## 没有安全框架时

每个接口都要手动检查登录状态：

```java
@PostMapping
public String add(@Valid @RequestBody User user, HttpSession session) {
    if (session.getAttribute("user") == null) {
        return "请先登录";  // 每个方法都要写
    }
    userService.add(user);
    return "新增成功";
}
```

接口多了以后，这段判断散落在几十个方法里，改起来是噩梦。

## Spring Security 做了什么

它在你的 Controller **前面**放了一堵墙——所有请求先经过这堵墙，墙判断放行后，请求才到达 Controller。

```
请求 → Security 过滤器链 → 认证？授权？→ Controller
                        ↓
                    拒绝（401/403）
```

## 核心概念：认证 vs 授权

这两个词经常被混用，但它们是**两个不同的问题**：

| | 认证（Authentication） | 授权（Authorization） |
|---|---|---|
| 问题 | 你是谁？ | 你能干什么？ |
| 时机 | 先认证 | 再授权 |
| 失败 | 401 Unauthorized | 403 Forbidden |
| 例子 | 输入用户名密码登录 | admin 能删用户，user 不能 |

## 三个核心组件

配置 Spring Security，本质上就是定义三个 Bean：

### 1. PasswordEncoder — 密码怎么存

**绝对不能存明文密码。** `BCryptPasswordEncoder` 用 BCrypt 算法做单向哈希，即使数据库泄露，密码也解不出来。

```java
@Bean
public PasswordEncoder passwordEncoder() {
    return new BCryptPasswordEncoder();
}
// "admin123" 编码后 → "$2a$10$NqGqJ0Yy..." （不可逆）
```

### 2. UserDetailsService — 用户从哪来

告诉 Spring Security 去**哪里查用户**。学习阶段用内存用户，生产环境连数据库。

```java
@Bean
public UserDetailsService userDetailsService() {
    var admin = User.builder()
            .username("admin")
            .password(passwordEncoder().encode("admin123"))
            .roles("ADMIN")
            .build();
    return new InMemoryUserDetailsManager(admin);
}
```

### 3. SecurityFilterChain — 哪些路径要保护

定义**什么路径需要什么权限**。

```java
@Bean
public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
    http
        .authorizeHttpRequests(auth -> auth
            .anyRequest().authenticated()   // 默认全部需要登录
        )
        .httpBasic(withDefaults())   // 用 HTTP Basic 认证（用户名:密码 Base64）
        .csrf(AbstractHttpConfigurer::disable);  // REST API 不需要 CSRF
    return http.build();
}
```

**安全原则**：SecurityConfig 做全局兜底（`.anyRequest().authenticated()`），个别接口的权限用 `@PreAuthorize` 在方法上声明：

```java
@GetMapping
@PreAuthorize("permitAll()")          // 仅这个接口公开
public Result<Page<User>> list(...) { ... }

@DeleteMapping("/{id}")
@PreAuthorize("hasRole('ADMIN')")     // 仅 ADMIN 角色能访问
public Result<Void> delete(...) { ... }
```

需要 `@EnableMethodSecurity` 开启方法级权限控制。
```

**为什么关 CSRF？** CSRF 防护是为了防止别人伪造浏览器表单提交。REST API 不依赖浏览器 Cookie，用 CSRF 反而多一层麻烦。

## HTTP Basic 认证

配置了 `httpBasic()` 后，客户端在请求头里带用户名密码：

```
Authorization: Basic YWRtaW46YWRtaW4xMjM=
                         ↑ admin:admin123 的 Base64
```

Postman 测试：Auth 标签 → Basic Auth → 填 admin / admin123。

## 测试验证

重启应用后：

```
GET  /user           → 401 Unauthorized（没认证）
GET  /user           → 200 OK（带 Authorization 头）
POST /user           → 401 Unauthorized（没认证）
POST /user           → 200 OK（带 Authorization 头）
```

## 为什么用内存用户而不是连数据库

学习阶段先理解 Security 的**机制**（过滤器链、认证流程、权限模型），用户存在哪是**实现细节**。用 `InMemoryUserDetailsManager` 可以零干扰地理解核心概念。后面换成数据库，只改 `UserDetailsService` 的 Bean，其他地方不动。

## 架构简图

```
SecurityConfig
  ├── PasswordEncoder        ← BCrypt，单向加密
  ├── UserDetailsService     ← 从内存/数据库加载用户
  └── SecurityFilterChain    ← 拦截规则
        ├── authorizeHttpRequests  → 哪些路径要什么权限
        ├── httpBasic              → 用 HTTP Basic 认证
        └── csrf                   → REST API 关掉
```
